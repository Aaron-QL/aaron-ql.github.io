但是对宿主机的lo设备抓包却看不到任何信息，这并不符合我的预期，目前还没搞明白是什么原理。

测试结论：

透明代理不跟客户端或服务端在同一节点的情况下能够正常运行。

客户端（在容器内）和正向代理在同一节点时流量无法被代理劫持。

遇到的问题

  • 宿主机上创建的server访问不通：防火墙进行了限制，关闭后结局
  • netns1内ping不通baidu.com：同样是防火墙搞的鬼，它删除了我添加的iptables规则，关闭后解决
  • 对于通过docker创建的容器，它所属的网络的网络命名空间无法通过ip netns命令看到：因为 docker 把网络命名空间的文件存在另一个位置，没有创建所需的符号链接(symbolic link)
  • veth0接到交换机上时链路不通：需要去掉veth0的IP
  • 发往本地的数据包走lo不走eth0
  • 一次HTTP请求与响应会产生10或11个包（三次握手（3）+请求与响应（4）+四次挥手（3或4）
  • 代理和被代理程序不能在同一个节点
  • 宿主机上抓lo设备看不到信息
  • TProxy如何做Outbound方向上的代理
  • nat的方式会改写数据包，服务端程序内看到就是监听端口，看不到原始端口
  • 对于一次HTTP请求，iptables记录的REDIRECT pkts数量只+1，虽然传输了多个数据包
  • iptables的-i和-o选项在路由前筛选设备的逻辑
  • 数据包是否会完整经过iptables的每个链？
  • 监听范围端口是否可行？
